MuJoCo separates a simulation into two data structures with sharply different roles. **`mjModel`** is the compiled, constant description of the system produced by parsing an MJCF file (or loading a binary `.mjb`). It holds all fixed sizes (`nq`, `nv`, `nbody`, `njnt`, `nu`, ...), the tree topology, inertial parameters, geom and mesh data, actuator/tendon/constraint definitions, and solver options. Because it never changes during stepping, one `mjModel` can be shared read-only across many threads. **`mjData`** is the mutable per-instance workspace: it stores the physical state and every quantity derived from it, and it owns a stack/arena for temporary allocations whose size can vary step to step (for example, with the number of active contacts).

The complete physical **state** is the tuple \((\text{time},\ \texttt{qpos},\ \texttt{qvel},\ \texttt{act})\), where `act` is the vector of actuator activation states. Given that state and the controls (`ctrl`, plus optional `qfrc_applied` and Cartesian `xfrc_applied`), `mjData` is filled with derived results: global body poses (`xpos`, `xquat`), the joint-space mass matrix \(M\), bias forces, the detected contact list, constraint Jacobians and forces, and the resulting acceleration `qacc`. Nothing in this derived set is independent input; it is all a deterministic function of state and control, which is what makes runs reproducible.

One integration step, `mj_step`, is a two-phase operation: compute the acceleration by **forward dynamics**, then advance the state by the chosen integrator. Forward dynamics itself is decomposed into ordered stages so that intermediate results can be reused. The **position** stage runs forward kinematics, builds the mass matrix via the composite-rigid-body algorithm, computes tendon lengths/moments, and performs collision detection to populate contacts and constraint Jacobians. The **velocity** stage computes body velocities and the bias force \(c(q,v)\) via recursive Newton-Euler, plus passive spring/damper/fluid forces. The **actuation** stage turns `ctrl` and `act` into `qfrc_actuator`. The **acceleration/constraint** stage then solves for `qacc` and the constraint forces together.

The core equation solved each step is the constrained forward dynamics

\[ M(q)\,\dot v = \tau_{\text{applied}} - c(q, v) + J^{\top} f, \]

where \(f\) is the constraint force returned by the solver. After `qacc` is known, the integrator advances the state over the fixed time step \(h = \texttt{option.timestep}\). Available integrators include semi-implicit **Euler** (default), **RK4** (fourth-order explicit), and **implicit**/`implicitfast` schemes that treat velocity-dependent forces implicitly for stability with strong damping. Quaternion components of `qpos` are advanced on the rotation manifold rather than by componentwise addition.

A distinctive consequence of this pipeline is exact **inverse dynamics**. Because the constraint model is invertible, `mj_inverse` can recover the forces that produced an observed acceleration and, run back-to-back with forward dynamics, reproduces the inputs to numerical precision. This invertibility, together with the deterministic model/data split, is what makes the engine attractive for model-based control, trajectory optimization, system identification, and reinforcement learning, where derivatives and repeatable rollouts matter as much as raw stepping speed.
