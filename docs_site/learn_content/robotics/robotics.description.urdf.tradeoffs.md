Three XML formats dominate open robot modeling, and choosing among them is really a choice about *what a description is allowed to say*. The Unified Robot Description Format targets a single robot as a kinematic **tree** of links and joints with minimal dynamics. The Simulation Description Format was designed as a superset for whole *worlds*: it can hold multiple models, lights, terrain, physics settings, sensors, and plugins, and it permits closed kinematic loops. The physics-engine-native format used by high-performance simulators (an XML with nested `<body>` elements, generalized coordinates, and a rich constraint/actuator model) is oriented toward fast, stable contact dynamics. None is strictly better; each encodes the assumptions of the toolchain that produced it.

## Where they diverge

The first axis is **topology**. A tree-only format forbids cycles, so parallel and closed-loop mechanisms cannot be expressed directly; the world-oriented and physics-native formats both close loops, one by declaring a joint between two already-connected links, the other by adding an equality/`connect` constraint that ties two bodies. The second axis is **frame semantics**. The tree format fixes each joint's `<origin>` as the parent-to-child transform, so a link's frame is implied by its incoming joint; the physics-native format nests bodies and gives each a pose relative to its parent in generalized coordinates; the world format lets a `<pose>` be expressed relative to an arbitrary named frame, which is more flexible but also more error-prone. A third axis is **reuse and defaults**: the physics-native format has class-based `<default>` inheritance built in, whereas the tree format has none and relies on an external macro preprocessor. Finally, **physics fidelity** differs sharply, from a bare mass/inertia/damping model, through plugin-driven sensors and solvers, to first-class contact-solver parameters, tendons, soft constraints, and a detailed actuator model.

## Round-trip and where information is lost

Conversion is routinely one-way and **lossy**, so it is best treated as a compilation step rather than a reversible transform. Converting a tree description into a world or physics-native model is common and well supported, because the target is a superset in structure; the reverse direction discards whatever the tree cannot hold. The lossy categories are predictable: closed loops collapse or must be re-added by hand, sensors and actuators are dropped or approximated, visual/material richness is flattened, solver and contact parameters have no home, multiple models and environment elements disappear, and relative-frame poses must be baked into fixed parent-to-child transforms. A practical rule is to keep the authored source in whichever format is most expressive for your workflow and generate the others, rather than editing the derived artifacts and trying to merge changes back.

## Choosing deliberately

Pick the tree format when the deliverable is a single manipulator or mobile base that must interoperate with the widest set of planners and visualizers, and when a macro layer can supply the parameterization it lacks. Prefer the world-oriented format when the scene, not just the robot, is the object of interest, or when the mechanism genuinely has loops. Reach for the physics-native format when contact-rich dynamics, controller-in-the-loop training, or high step rates dominate, and accept that its constraint and actuator constructs will not survive a trip back to a tree. Because each format bakes in a different notion of frames, topology, and physics, treat interchange as a designed pipeline with explicit, tested conversions and a documented record of what each hop drops.
