In reinforcement learning (RL) a learning agent improves its behavior by interacting with an **environment**: the environment defines *the world*, and the agent defines *the policy* that acts within it. The two are deliberately separate. A physics simulation, a kinematic model, or a piece of instrumented hardware supplies the environment, i.e. everything outside the decision-maker: the dynamics that evolve state over time, the sensory signals the agent observes, and the scalar reward that scores its choices. The learning algorithm, whatever it is, only supplies the policy. Getting this boundary right is the first design task, because a well-posed environment can be reused across many algorithms, while a leaky one (rewards computed from privileged state the agent could never observe, hidden nondeterminism) quietly corrupts every experiment run against it.

## The Markov decision process

The standard formalization is the **Markov decision process** (MDP), the tuple \((\mathcal{S}, \mathcal{A}, P, R, \gamma)\): a state space \(\mathcal{S}\), an action space \(\mathcal{A}\), a transition kernel \(P(s' \mid s, a)\) giving the probability of the next state, a reward function \(R(s, a, s')\), and a discount factor \(\gamma \in [0, 1)\). At each step the agent observes a state \(S_t\), selects an action \(A_t\), and the environment returns a reward \(R_{t+1}\) and a new state \(S_{t+1}\). The agent's objective is to maximize the expected **discounted return**

\[ G_t = \sum_{k=0}^{\infty} \gamma^{k} R_{t+k+1}. \]

The defining assumption is the **Markov property**: the distribution of the next state and reward depends only on the current state and action, not on the full history. This is what makes the state a *sufficient statistic* for decision-making and is why environment design so often reduces to "choose a state representation that is actually Markov."

## What an environment must provide

Concretely an environment exposes three things: a way to **reset** to an initial state (sampled from a start distribution), a way to **step** the world forward given an action, and definitions of its **observation space** and **action space**. The step operation returns the next observation, the reward, and a signal indicating whether the episode has **terminated** (reached a goal or failure state) or been **truncated** (hit a time limit). Rewards are the only channel through which task intent is communicated, so their design (dense shaping versus sparse goal signals, penalties for effort or constraint violation) largely determines what the agent learns.

When the agent cannot see the full state (occlusion, sensor noise, latency), the problem is a **partially observable MDP** (POMDP): the agent receives observations \(O_t\) drawn from an emission model \(O(o \mid s)\) rather than \(S_t\) itself, and must infer state from a history of observations. Most real robotic tasks are POMDPs even when we model them as MDPs for tractability, which is one reason the choice of what to include in the observation vector is a first-class modeling decision rather than an afterthought.

## Why the separation matters in engineering practice

Because the environment is where physics and task semantics live, it must be **deterministic given a seed** and reproducible: identical seeds and identical action sequences must reproduce identical trajectories, or regression testing and fair algorithm comparison become impossible. High-throughput training also relies on running many environment copies in parallel (vectorized or distributed rollouts) to decorrelate samples and saturate compute, which requires the environment to be cheap to instantiate and free of shared global state. For robotics specifically, the environment typically wraps a rigid-body dynamics engine, so its fidelity, its contact and friction models, actuator limits, and integration timestep, directly bounds how well any learned policy will transfer to hardware, a concern treated in depth under sim-to-real transfer.
