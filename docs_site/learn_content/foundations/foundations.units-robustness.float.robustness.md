Geometric algorithms rest on a small number of **predicates**: yes/no or sign queries that classify a configuration. The two archetypes are the **orientation** test (does point \( c \) lie left of, right of, or on the directed line through \( a \) and \( b \)?) and the **incircle** test (is point \( d \) inside, outside, or on the circle through \( a \), \( b \), \( c \)?). Each reduces to the sign of a determinant, for orientation in the plane:

\[ \mathrm{orient2d}(a,b,c) = \operatorname{sign}\begin{vmatrix} a_x - c_x & a_y - c_y \\ b_x - c_x & b_y - c_y \end{vmatrix}. \]

The answer feeds combinatorial decisions: which side of a face a point falls on, whether two edges cross, how a mesh is triangulated. The output is discrete, but it is computed from continuous coordinates through floating-point arithmetic.

## The robustness problem

Herein lies the trouble. When the true determinant is near zero (points nearly collinear or cocircular), rounding error can flip the computed sign, so the predicate lies. Unlike a numerical result that is merely slightly inaccurate, a *wrong sign* is a categorical error: it produces contradictory answers, for example concluding that \( a \) is left of \( bc \) while also \( b \) is left of \( ca \) and \( c \) is left of \( ab \), a configuration no real point set can produce. Downstream code that assumes consistency then builds impossible topology, loops forever, or crashes. In a boundary-representation (B-rep) modeler this manifests as non-manifold edges, faces that fail to close, or a Boolean operation that silently yields an invalid solid. The deep issue is a mismatch between **numeric geometry** (approximate coordinates) and **combinatorial topology** (which must be exactly self-consistent).

## Exact predicates and adaptive precision

The robust remedy is to evaluate the *sign* exactly even though coordinates are floating-point. The key observation is that the determinant of numbers with finite binary expansions is itself an exact rational (indeed a finite-precision) quantity, so it can in principle be computed without error. Doing so naively with arbitrary-precision arithmetic is correct but slow. The practical approach, due to Shewchuk, is **adaptive-precision** evaluation guarded by a **floating-point filter**: first compute the predicate in fast hardware arithmetic together with a rigorous error bound; if the approximate value exceeds that bound in magnitude, its sign is certified and returned immediately, and only in the rare near-degenerate cases does the code fall back to progressively more exact expansions until the sign is guaranteed. Because near-degenerate inputs are uncommon, the average cost stays close to the naive floating-point version while the answer is always correct.

Two further techniques complete the toolkit. **Interval arithmetic** provides an alternative filter, carrying a validated lower and upper bound so that a sign is trusted only when the interval excludes zero. And exact degeneracy, points that are *genuinely* collinear or cocircular, can be resolved consistently by **symbolic perturbation** schemes such as Simulation of Simplicity, which imagine an infinitesimal, systematically ordered displacement of each input so that no true zero ever occurs, removing special-case handling while keeping all decisions mutually consistent. Together, exact predicates, filters, and perturbation let a geometry kernel enjoy the speed of floating-point while guaranteeing the logical consistency that topological algorithms demand.
