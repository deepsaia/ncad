A **tool library** is the registered, reusable catalog of cutting tools available to a process plan. Rather than describing a cutter inline in each operation, the plan references a library entry by identity; the library owns the tool's geometry, cutting parameters, and identifiers. This indirection is what makes plans portable and shop-consistent: the same operation can be re-costed or re-posted against a different physical tool by rebinding the reference, and a tool's data lives in exactly one place.

Each library record is a structured, parametric description. The **geometry** block carries diameter, corner radius, taper, tip angle, flute count, helix angle, flute length, and overall length; these are the same parameters the toolpath and simulation consume for envelope and gouge checking. A **cutting-data** block associates recommended surface speed, feed per tooth, and depth-of-cut ranges, often keyed by workpiece material, so that feeds and speeds can be looked up rather than re-derived. Administrative fields include a unique tool identifier, a description, and, on machined jobs, the **pocket, turret station, or magazine index** where the physical tool lives, plus a tool-number that the posted program calls with a tool-change command.

The motive for a standardized library is data exchange. Cutting-tool data is generated by tool vendors, consumed by planning and simulation software, and reconciled by tool-management and ERP systems; a system-independent representation lets these exchange tool definitions without lossy re-keying. The relevant international standard defines a neutral information model and a dictionary of tool and adaptive-item properties precisely so a diameter or a corner radius means the same thing across every system that reads the record.

A subtlety is that the useful unit is rarely the bare cutter. A library commonly stores a **tool assembly**: the cutting item plus its holder, plus any extension or collet, resolved into a single overhang and profile. That assembled geometry is what drives collision and reach checks (see tool holders and gauge length), while the cutting item alone drives chip-formation math. Keeping the library as a first-class, versioned resource, with stable IDs and explicit units, is what lets a deterministic plan reproduce identical output every time it is rebuilt against the same catalog.
