Posted G-code is emitted with the intention of cutting metal, but running unverified code risks collisions, gouges, and scrap. **Back-plotting** is the practice of reading a program back, interpreting each block exactly as a controller would, and reconstructing the resulting tool motion, either as a wireframe of the tool-center path or as a full material-removal simulation. It closes the loop: the same block stream the machine will receive is fed to an interpreter that reproduces the controller's semantics, so the geometry the code *actually* produces can be inspected before it produces anything physical.

The foundation of back-plotting is an interpreter faithful to the language's execution model, the same modal groups, order-of-execution rules, offset arithmetic, and canned-cycle expansions described by RS274/NGC. Motion blocks are decoded into their true paths: linear moves are straight segments, and \(G2/G3\) arcs are reconstructed in the active plane. An arc word supplies either a radius or center offsets \(I\,J\,K\); the swept path is parameterized as
\[
\mathbf{p}(\theta) = \mathbf{c} + R\,[\cos\theta,\ \sin\theta], \qquad \theta \in [\theta_0, \theta_1],
\]
with the sweep direction set by clockwise versus counterclockwise selection. Because the back-plotter must apply work offsets, tool-length offsets, and cutter compensation just as the controller does, it inherently exercises the very parts of the post that are hardest to get right, making it a verification instrument as much as a visualization.

## Levels of verification

Verification spans a spectrum of fidelity. The cheapest level is *path back-plot*, drawing the tool-center trajectory and coloring rapids versus feeds, which catches gross errors like a missing retract, an inverted arc, or a plunge at rapid rate. A richer level is *material-removal simulation*, where the tool swept-volume is subtracted from a stock model so the emerging part can be compared against the design; this catches gouges, uncut regions, and incorrect stock allowances. The most rigorous level adds *collision and machine-envelope checking*, testing the tool holder, spindle, and fixtures against travel limits and against each other, which is essential for multi-axis motion where intuition fails.

A disciplined design distinguishes *simulating the toolpath* (does the strategy remove the right material?) from *verifying the posted G-code* (does the emitted block stream reproduce that strategy on the target dialect?). The two can diverge precisely because the post transforms the path, applying compensation, expanding cycles, reformatting arcs, so verifying the final code, not just the pre-post toolpath, is what actually protects the machine. This is also why deterministic, golden-file posting pairs naturally with back-plotting: a stored reference program plus a back-plot regression makes any change in emitted geometry visible and reviewable.

Back-plotting matters because the cost asymmetry is severe: a few seconds of simulation weighed against a wrecked spindle, a broken cutter, or a scrapped casting. It is the last independent check between abstract intent and irreversible physical action, and treating it as a routine, automated gate, rather than an optional convenience, is a hallmark of professional NC workflows.
