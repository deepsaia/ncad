Numerical-control (NC) programs are written in a line-oriented language whose statements are called *blocks*. Each block is a sequence of *words*, and each word is a single letter *address* immediately followed by a real-number *value*. The address letter names a channel of meaning; the value parameterizes it. This word-address grammar traces to the EIA RS-274 standard and its later refinement RS274/NGC, and it remains the lingua franca that a *post-processor* emits so that a controller can drive the machine. Because the format is textual and modal, a program is compact, human-readable, and streamable one block at a time.

The five workhorse address families cover distinct roles. **G** words (*preparatory*) set the geometric and modal intent of motion, for example rapid positioning \(G0\), linear feed \(G1\), and clockwise/counterclockwise circular interpolation \(G2/G3\); coordinate-plane, units, and distance-mode selects (\(G17/G18/G19\), \(G20/G21\), \(G90/G91\)) also live here. **M** words (*miscellaneous* or machine) switch discrete machine functions such as spindle on/off (\(M3/M4/M5\)), coolant (\(M7/M8/M9\)), tool change (\(M6\)), and program end (\(M2/M30\)). **T** selects a tool number, **S** sets spindle speed, and **F** sets feed rate. The coordinate words \(X\,Y\,Z\) (plus \(A\,B\,C\) rotary and \(I\,J\,K\) arc-center or vector offsets) supply the numeric geometry that the G word interprets.

## Interpretation order and semantics

A critical subtlety is that a block is **not** executed left to right. RS274/NGC defines a fixed order of execution independent of word order within the block: settings such as feed rate and spindle speed are established, then tool-length and cutter-compensation modes, then the commanded motion, and finally certain stopping functions. Consequently `G1 X10 F200` and `F200 X10 G1` are identical. The interpreter first parses every word in the block, resolves the active *modal groups* (only one G word from each mutually exclusive group may appear per block), evaluates any expressions, and only then acts.

Spindle speed \(S\) is normally expressed in revolutions per minute, while feed \(F\) is expressed as distance per unit time in feed-per-minute mode (or distance per revolution in feed-per-revolution mode). For a milling cutter of diameter \(D\) the relationship between the *cutting speed* \(v_c\) at the tool periphery and spindle speed \(n\) is
\[
n = \frac{v_c}{\pi D}, \qquad v_f = n\, f_z\, z,
\]
where \(v_f\) is the table feed, \(f_z\) the feed per tooth, and \(z\) the number of teeth. A post-processor computes these values from the machining strategy and material and emits them as \(S\) and \(F\) words.

The vocabulary matters because it is the contract between geometry-aware CAM logic and dumb, real-time motion hardware. Everything a toolpath *means*, safe retract heights, arc directions, unit systems, spindle state, must survive the reduction to letters and numbers with no ambiguity, because a controller has no model of the part. Getting the vocabulary and its modal semantics exactly right is what separates a program that cuts the intended part from one that gouges it.
