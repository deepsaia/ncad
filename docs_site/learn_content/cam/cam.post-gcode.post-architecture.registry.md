Although RS274/NGC defines a common core, no two controller families speak exactly the same dialect. They differ in program header and footer conventions, block numbering, permissible G/M codes, canned-cycle support, decimal formatting, maximum feed rates, arc conventions (center vectors \(I\,J\,K\) versus radius \(R\)), and whether features such as high-speed look-ahead or subprograms are available. A **post-processor** is the software component that translates a machine-neutral, geometric *toolpath* into the exact block stream a specific controller will accept. The reason posts exist at all is that the CAM layer should reason about geometry and machining strategy, not about a particular vendor's quirks; the post is the adapter that absorbs those quirks.

At scale, the sound architecture is a **registry of pluggable dialects** layered over one shared generic core. The core owns everything dialect-independent, motion-block assembly, modal-state tracking, coordinate formatting, arc decomposition, and the ordering rules of RS274/NGC. Each dialect is a plug-in that overrides only what differs: the program prologue/epilogue, the tool-change sequence, which canned cycles to emit versus expand long-hand, number formatting, and any nonstandard codes. This is the classic strategy/template-method separation: the invariant algorithm lives once, and variation is injected through a small, well-defined interface. A registry keyed by machine identifier lets the system select the right dialect at post time without conditionals scattered through the core.

## Why a generic core plus overrides

Centralizing the invariant logic prevents the combinatorial explosion and drift that occur when every machine gets a hand-copied, independently maintained post. A bug in arc handling or modal tracking is fixed once in the core and every dialect benefits; a new machine is onboarded by supplying a thin override rather than a whole program. The interface between core and dialect should be narrow and explicitly typed, formatting hooks, cycle-capability flags, header/footer emitters, so that a dialect cannot silently corrupt the core's state machine. Capability negotiation (does this controller support \(G83\)? radius arcs? \(G54.1\) extended offsets?) is data the core queries from the dialect and adapts to, degrading gracefully by expanding unsupported cycles into primitive moves.

Determinism is a first-class requirement of this architecture. Given the same toolpath and the same dialect, a post must emit byte-identical output every run; otherwise verification, diffing, and regression testing become impossible. That argues for a pure translation stage with no hidden state or nondeterministic ordering, and it makes *golden-output* tests (a fixed toolpath posted through a fixed dialect compared against a stored reference file) the natural regression net for the whole subsystem.

The payoff of the registry model is operational: a shop can maintain dozens of machines from a single CAM environment, add or retire controllers by editing a registry entry, and trust that the shared geometric and modal logic behaves identically everywhere. It cleanly separates *what to machine* (owned by CAM), *how RS274/NGC works* (owned by the core), and *what this controller demands* (owned by the dialect), which is exactly the layering that keeps a post-processing system correct as the machine fleet grows.
