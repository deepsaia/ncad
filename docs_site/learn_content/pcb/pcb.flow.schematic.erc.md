The **Electrical Rule Check (ERC)** is a static verification pass over a schematic's connectivity, analogous to a compiler's type checker for a program. Where a Design Rule Check (DRC) inspects *physical* geometry on the board, ERC inspects the *logical* circuit: it asks whether the way pins have been wired together is electrically sensible before any copper is drawn. Because errors caught here cost minutes to fix but the same errors caught after fabrication cost a board respin, ERC sits at a high-leverage point in the flow, run repeatedly as the schematic matures.

The core idea is a **pin-type model**. Every pin is tagged with an electrical role, typically drawn from a small vocabulary: *output*, *input*, *bidirectional*, *tristate*, *passive*, *power-in*, *power-out*, *open-collector/open-emitter*, and *no-connect*. For each net (each set of pins declared electrically common) the checker examines the multiset of pin types attached to it and evaluates them against a compatibility matrix. Conceptually the rule is a symmetric predicate over pairs,

\[ \text{conflict}(net) = \bigvee_{\,i<j\,\in\, net} \; M\!\left(t_i, t_j\right), \]

where \(t_i\) is the type of pin \(i\) and \(M\) returns *error*, *warning*, or *ok*. Classic flagged conditions are two *outputs* driving the same net (contention), an *input* pin with no driver, a *power-in* pin that is never fed by a *power-out* or supply, and any pin left completely unconnected. A pin explicitly marked *no-connect* suppresses the "floating pin" complaint, which is why deliberate no-connect flags are themselves part of good schematic hygiene.

Beyond pin typing, mature ERC engines add checks that a human reviewer would otherwise perform by eye: duplicate reference designators, nets with only a single pin (a wire that connects nothing), overlapping or dangling wire endpoints that *look* connected but are not, mismatched bus membership, and power nets that lack a corresponding power-supply or `PWR_FLAG`-style source declaration. Each violation is emitted as a structured issue with a severity and a location, so the results are data to be triaged rather than a hard pass/fail gate.

The practical value of ERC is that it converts a class of silent mistakes into loud, early, addressable ones. It cannot prove the circuit *works* (it knows nothing of voltage margins, timing, or intent), but it guarantees the connectivity is internally consistent and free of the mechanical slips (a mis-drawn wire, an unpowered part, a shorted output) that dominate real schematic bugs. A schematic that passes a well-configured ERC with all warnings deliberately reviewed is the correct precondition for extracting the netlist and moving into layout.
