A mechanical/parametric CAD system has no business re-implementing schematic capture, copper autorouting, electrical design-rule checking, or field solvers -- those are deep specialities with their own mature tools. The pragmatic architecture is a **delegation seam**: the mechanical side owns the *physical envelope* (board outline, mounting holes, keep-outs, connector positions, height zones) while a dedicated electrical-CAD tool owns the *electrical design* (nets, traces, stackup, physics-based DRC). The two collaborate by **reading and writing a shared native board file** rather than by one tool absorbing the other. The board file becomes the contract at the seam.

## Round-trip through a native board format

Concretely, this means importing the electrical tool's board file to obtain the current outline, layer stackup, and component placements as geometry, and writing back the mechanical constraints the electrical designer must respect. The reference open-source ECAD board file (`.kicad_pcb`) is an **S-expression**: a Lisp-style, human-readable tree of nested lists such as `(module ...)`, `(gr_line ...)`, and `(net ...)`. That text-based structure is significant for engineering hygiene -- it diffs cleanly and versions well under source control, unlike opaque binary CAD files -- which makes a read/write bridge tractable and reviewable.

## Ownership and the lossy-round-trip hazard

The hard part of any round-trip is deciding **who is authoritative for each field**, because a naive write-back can clobber data the other tool cares about. A disciplined seam is *asymmetric*: the mechanical side may author and update board outline, mounting-hole positions, keep-out regions, and connector placement, but must treat nets, traces, and copper pours as read-only and preserve them verbatim on write. The failure mode to guard against is **lossy round-tripping** -- silently dropping or reordering electrical entities the mechanical model does not understand -- so a correct implementation copies through unknown or non-owned nodes untouched rather than regenerating the whole file from its own partial model.

## Why this pattern generalizes

This is an instance of a broader interchange principle: prefer **bidirectional exchange through a well-defined, tool-native format** with a single source of truth per data category, over monolithic re-implementation or a fragile export-only pipeline. It keeps each discipline in the tool that does it best, lets the electrical fab/assembly outputs (Gerber, drill, netlist, assembly data) continue to be generated by the tool that owns the copper, and confines the mechanical CAD system to what it is genuinely better at -- 3-D form, fit, and the physical constraints that flow back into the electrical layout.
