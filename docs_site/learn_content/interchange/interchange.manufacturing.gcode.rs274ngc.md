G-code is the low-level command language that tells a computer-numerical-control (CNC) machine where to move, how fast, and what auxiliary functions to switch on. RS274NGC is the dialect specified by the U.S. National Institute of Standards and Technology (NIST) for its Enhanced Machine Controller (EMC) project; it formalized, in a public document, the informal RS-274 conventions that grew out of the original EIA/RS-274-D standard and later ISO 6983. Because it is openly published and precisely defined, RS274NGC is the de facto reference dialect against which most open and many proprietary controllers are compared. A program is a sequence of ASCII lines called *blocks*, each a set of *words*: a letter address (G, M, X, F, S, T, ...) followed by a real number.

## Words, modes, and the interpreter

The language is *modal*: a commanded state persists until it is explicitly changed. Setting the motion mode to linear feed (`G01`) makes every subsequent block with coordinates a straight cut until another motion word overrides it. Modes are organized into *modal groups* (motion, plane selection, distance mode, units, feed-rate mode, spindle, coolant, and so on), and it is an error to put two words from the same group in one block. Within a single block the interpreter does **not** execute words left to right; it parses the whole line, then applies items in a fixed order of execution (comments, feed-rate mode, feed and spindle set, tool select, tool change, coordinate-system and plane selection, cutter compensation, then motion, then stopping). Understanding that fixed order is essential to predicting what a block actually does.

## Coordinated motion and interpolation

The core service is *coordinated multi-axis motion*: the axes start and finish together so the tool follows the intended path. For linear moves the controller drives the joints so the tool tip traces the segment from the current point to the target while holding the programmed feed rate \(F\) as the tangential speed along the path. For circular interpolation (`G02` clockwise, `G03` counterclockwise) an arc is defined either by a radius word `R` or by the arc-center offsets `I`, `J`, `K` measured from the start point in the active plane (`G17` XY, `G18` XZ, `G19` YZ). The center offsets over-determine the arc, so a conforming interpreter checks consistency: the distance from the start point to the center must match the distance from the end point to the center within a tolerance, otherwise the block is rejected. The tangential feed along an arc of radius \(r\) relates to angular rate by

\[ v = \lVert \dot{\mathbf{p}} \rVert = r\,\lvert\dot{\theta}\rvert = F. \]

## Compensation, offsets, and canned cycles

Raw toolpaths are written to the desired part edge, but a physical tool has finite radius and length. *Cutter radius compensation* (`G41`/`G42`) offsets the commanded path to one side by the tool radius stored in a table, so the same program cuts correctly as tools wear or are replaced. *Tool length offset* (`G43`) shifts the Z origin by a measured length. *Work coordinate systems* (`G54`-`G59`) let a program be written in part coordinates while the setup defines where the part sits on the table. *Canned cycles* (drilling, peck-drilling, tapping, boring; `G81`-`G89`) encapsulate a repeated motion pattern into one modal command, reducing a multi-line hole sequence to a single block per hole.

## Why it matters and its limits

G-code is the near-universal handoff between CAM and the machine, which is exactly its strength and its weakness. It is portable at the level of geometry-plus-motion, but it is *procedural and post-processed*: it describes moves, not intent. The program is generated for one machine kinematics, one tool set, and one controller dialect through a *post-processor*, and dialects differ in canned cycles, subprogram syntax, and M-code meanings (only a handful, such as `M03`/`M04`/`M05` spindle and `M06` tool change, are broadly consistent). There is no notion of the feature being made, the stock, the tolerances, or why a value was chosen, so a G-code file cannot be safely re-optimized or re-targeted to a different machine without going back to the CAM model. Those limitations are precisely what the feature-based STEP-NC model set out to address.
