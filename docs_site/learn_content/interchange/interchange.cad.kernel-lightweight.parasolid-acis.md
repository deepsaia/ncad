Most solid-modeling applications are built on a **geometric modeling kernel**, the library that owns the mathematical B-rep, computes booleans, fillets, and offsets, and enforces topological validity. Two kernels have dominated the mechanical CAD landscape, and each has a native serialization that has become a de-facto interchange format: the Parasolid *transmit* file (`.x_t` text, `.x_b` binary) and the ACIS *save* file (`.sat` Standard ACIS Text, `.sab` binary). Because a large fraction of commercial modelers embed one of these kernels, writing that kernel's own transmit format is often the highest-fidelity way to move a model, the receiving system deserializes the exact same entity records its kernel would have produced internally, with no translation step in between.

## Why kernel-native beats neutral, and where it doesn't

A neutral standard such as STEP must map a kernel's geometry onto a common schema and back, and any concept the schema does not express, kernel-specific tolerances, blend recipes, or attribute bundles, can be lost or approximated. A kernel-native file avoids that mapping entirely: it is a direct save/restore of the modeler's data structures, so exact NURBS surfaces, oriented topology, and tolerant-modeling data (per-edge and per-vertex tolerances that let slightly imperfect geometry remain a valid solid) transfer without approximation. The cost is that these formats are **proprietary and version-locked**. The file carries a schema/version stamp, a newer kernel writes records an older reader cannot parse, and reading or writing them at all requires licensing the kernel. Neutral formats trade some fidelity for openness and longevity; kernel formats trade openness for exactness. Serious interoperability programs choose deliberately between the two.

## The boundary representation underneath

Both formats encode a **boundary representation**: a solid is a region of space bounded by an oriented, closed 2-manifold shell, decomposed into a topology graph, regions and shells, then faces, loops (`edge_loop`), edges, and vertices, with each face carrying a trimmed surface and each edge a curve. A kernel does not merely store this graph; it maintains its validity as an invariant across every operation. The governing constraint is the generalized **Euler-Poincare formula**:

\[ V - E + F - R = 2\,(S - G), \]

where \(V,E,F\) count vertices, edges, and faces, \(R\) counts inner (ring) loops that make holes in faces, \(S\) counts shells, and \(G\) is the genus (the number of through-holes/handles). Any modeling operation is expressible as a composition of Euler operators that preserve this relation, so a well-formed kernel file describes a topologically consistent solid by construction.

In practice these formats are requested as OEM interchange deliverables precisely because they are lossless within a kernel family, and they are avoided for archival because their proprietary, version-sensitive nature offers none of the multi-decade readability guarantees that a ratified, human-inspectable standard aims to provide. Understanding both sides of that tradeoff is the core of choosing an interchange strategy.
