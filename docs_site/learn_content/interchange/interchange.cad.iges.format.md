The **Initial Graphics Exchange Specification (IGES)** was the first widely adopted neutral format for moving CAD geometry between otherwise incompatible systems. It was published in 1980 by the U.S. National Bureau of Standards (now NIST) and stewarded through later revisions by the U.S. Product Data Association, culminating in Version 5.3 (1996), the last ratified edition. IGES exists because early CAD systems each had a private binary database; to share a model with a supplier or a downstream tool, the industry needed a documented, vendor-neutral, human-readable interchange. IGES became that lingua franca, and although STEP has largely superseded it for new work, IGES files remain common in legacy tooling, mold/die, and CAM pipelines.

## File structure

An IGES file is plain ASCII organized as fixed **80-column "card image" records**. Columns 73-80 of every line hold a section letter and a running sequence number, which is how the format survived the punched-card era and still parses unambiguously. A file is divided into ordered sections: an optional **Flag** section (for binary/compressed variants), then **Start (S)** (human-readable prologue), **Global (G)** (field/record delimiters, model units, resolution, scale, author and system identification), **Directory Entry (D)**, **Parameter Data (P)**, and **Terminate (T)**. The Directory and Parameter sections are the substance: every geometric or annotation object is an *entity* with a two-line, fixed-field directory record (entity type number, a pointer into the parameter section, and references to a transformation matrix, line font, level, view, and color) plus a free-format, comma-delimited parameter record holding the type-specific values.

## Entities and what IGES represents well

Each entity type has a numeric code: 100 circular arc, 110 line, 116 point, 124 transformation matrix, 126 rational B-spline curve, 128 rational B-spline surface, 142/144 (curve on a parametric surface and) trimmed parametric surface, 186 manifold solid B-rep object, and organizational entities such as 314 color, 402 associativity, and 408 singular subfigure instance for lightweight assembly-like referencing. The curve and surface primitives are the same rational B-splines used throughout CAD: a NURBS curve is

\[ C(u)=\frac{\sum_{i=0}^{n} N_{i,p}(u)\,w_i\,P_i}{\sum_{i=0}^{n} N_{i,p}(u)\,w_i}, \]

with control points \(P_i\), weights \(w_i\), and B-spline basis \(N_{i,p}\). Because the analytic and free-form curve/surface entities are mature, IGES is strong for wireframe and *surface* exchange, and the trimmed surface (type 144) is the workhorse for transferring skin geometry.

IGES's weaknesses define its niche. Solid support (the B-rep entity 186 and CSG entities) arrived late and is inconsistently implemented, so solids frequently degrade into disconnected surface "quilts" that require stitching on import. The format is essentially geometry-only: it has no semantic GD&T/PMI, only weak assembly expression via subfigures, and no configuration or provenance data. Those gaps, tolerance-driven leakage between surfaces, lost solid topology, and absent engineering intent, are exactly what motivated STEP and its application protocols to replace it for authoritative model exchange.
