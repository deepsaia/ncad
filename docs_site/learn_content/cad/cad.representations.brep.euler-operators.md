**Euler operators** are the low-level primitives that construct and modify a B-rep's *topology* while guaranteeing, by construction, that the result stays a valid closed manifold. Rather than let application code splice pointers into a half-edge structure by hand (where a single mistake yields a corrupt, non-closed body), a modeler exposes a small vocabulary of make/kill operations, each of which changes the entity counts in a way that provably keeps the topology consistent. Geometry is attached in a separate step, so these operators reason about connectivity alone.

Their correctness rests on the **Euler-Poincaré formula**, the topological invariant every well-formed manifold boundary must satisfy:

\[ V - E + F - R = 2\,(S - G) \]

where \(V\) vertices, \(E\) edges, \(F\) faces, \(R\) inner loops (rings/holes in faces), \(S\) shells, and \(G\) genus (number of through-holes/handles). For a simple cube \(8 - 12 + 6 - 0 = 2(1-0)\) holds. Each operator is named for the entities it makes and kills so that the invariant is preserved. The classical minimal set (in the tradition of Euler-operator solid modelers) includes **MVFS** (make vertex, face, shell: create an initial single-vertex shell), **MEV** (make edge and vertex), **MEF** (make edge and face by closing a loop), **KEMR** (kill edge, make ring: split a face's boundary into an inner loop), and **MEKR/MFKRH** style operators that create rings and change genus. Every operator has an exact inverse (its "kill" partner), and it is a theorem that any manifold boundary can be built up, and torn back down, purely by composing them.

The practical payoff is a **strict validity guarantee**: because the operator set is closed under the Euler-Poincaré relation, no legal sequence can ever produce a non-manifold, non-closed, or otherwise ill-formed shell. Higher-level features (extrude, sweep, fillet, boolean) are then implemented as *macros* over Euler operators plus geometry attachment, which is why the topological result of a feature can be trusted even before its surfaces are evaluated. Separating the combinatorial change (Euler operators) from the metric change (geometry) is also what makes robust modeling tractable: numerical trouble is confined to intersection and projection code, not to the graph surgery.

In a system built on a mature commercial kernel, Euler operators are typically **internal machinery** rather than an authoring surface: designers manipulate features and sketches, and the kernel translates those into the underlying make/kill sequences. Understanding them still matters, because the invariant they enforce is exactly what a validity checker asserts after every rebuild, and because reasoning in terms of "what did this feature do to \(V,E,F,R,S,G\)" is often the fastest way to diagnose a topology that has gone wrong.
