The concrete interface between an authoring tool and an external solver is a pair of file operations: **writing an input deck** and **reading back a result file**. The term *deck* is historical, dating to the era when a finite-element model was literally a deck of punched cards, and the vocabulary persists in the *card*-oriented, keyword-based formats many solvers still accept. Generating the deck means serializing the analysis model into the solver's own input language: a table of **nodes** (coordinates of the discretization points), a table of **elements** (connectivity that tells which nodes form each element and what element type it is), **material** and **section/property** definitions referenced by ID, **boundary conditions** and **load sets**, and the **solution or step** cards that tell the engine what analysis to run and with what controls. The deck must be internally complete and self-consistent, because the solver sees only the deck, not the rich model behind it.

Two disciplines dominate correct deck generation. The first is a **stable identity mapping**: every node, element, material, and set carries an integer or string ID, and those IDs must be assigned deterministically and preserved, because they are the *only* thing that will let results be mapped back onto model entities afterward. If the IDs shift between runs, the read-back is meaningless. The second is **unit consistency**. Most solvers are unit-agnostic: they multiply and add the numbers you give them and assume you have kept a coherent unit system (for example, consistent length, mass, time, and force such that the derived stress and energy units line up). The authoring layer is responsible for emitting numbers in one coherent system, since the deck itself usually carries no units.

On the return trip, the solver writes an **output/results file** containing the computed fields: nodal quantities such as displacement or temperature, and element or integration-point quantities such as stress, strain, and reaction forces. Read-back is the inverse of deck generation: parse those fields and re-associate them with model entities through the same ID map, so the analyst sees results *on the geometry* rather than as a raw table. A subtlety here is that different quantities live at different locations. Primary unknowns (displacements, temperatures) are naturally defined at nodes and are interpolated across an element by its shape functions,

\[ u(\boldsymbol{\xi}) = \sum_{i=1}^{n} N_i(\boldsymbol{\xi})\, u_i, \]

while derived quantities such as stress are computed most accurately at the interior **Gauss (integration) points**. To display a smooth stress field on the mesh, those integration-point values must be extrapolated to the element nodes and then averaged where elements share a node; this recovery step is a well-known source of visible artifacts and of the difference between raw and "averaged" stress plots.

## Robustness and provenance

Because the deck and the result file are the entire contract, the read side must be defensive. A solver run can fail (a singular stiffness matrix, non-convergence, a licensing or resource error), in which case there is no result file to parse and the correct behavior is to surface the solver's own log rather than to guess. Even on success, the reader validates that the returned node and element IDs match what was written, that all requested output blocks are present, and that the counts are consistent. Finally, the triple of *input deck + solver identity and version + result file* forms a reproducible provenance record: it is what lets an organization re-run, audit, or reproduce a result later, which is the whole point of keeping the model and the solve cleanly separated in the first place.
