# Bucket 0.2: Everyday Ops + Expressions: Design

**Status:** approved (brainstorm), ready for implementation plan.
**Track:** core spine (Phase 0, bucket 0.2). Builds directly on bucket 0.1.

## Goal

Grow the general spine from "sketch a rectangle and extrude it" to the everyday
mechanical-part vocabulary plus a parametric expression layer, so the boring bracket
(rectangle >> extrude >> 4 holes via an expression >> edge fillet) builds from a
parametric document and renders. **Gate:** the boring bracket builds from a parametric
document.

## Scope and interim decisions

References and selectors are bucket 0.3, so bucket 0.2 targets geometry with **explicit
parameters and a tiny fixed edge-keyword vocabulary**, not a query language:

- `hole` takes explicit `positions` on a plane; no patterns (Phase 3).
- `fillet`/`chamfer` take a named edge-set keyword (`all|top|bottom|vertical|horizontal`)
  resolved against the live solid's edges by orientation / z-position. Bucket 0.3
  replaces the keyword with real selectors over a versioned attribute model.
- `boolean` supports `cut|union|intersect` of two named features (target + tool); full
  multibody is Phase 3.
- Expressions cover `${ref}` + arithmetic now, plus the registered-function-call
  **mechanism** with a tiny stub library (proving design section 1's three-part model);
  a real engineering-function library is later.

## Architecture

Two independent additions on the existing spine; the `Builder`/executor loop is
unchanged. Ops still receive resolved params exactly as in bucket 0.1.

```
document (parameters + parts[features], some values are "${...}" strings)
        │
        ▼
 ParamResolver (new `params` unit): resolve parameters + inline every ${...}
   · restricted-AST evaluator (literals, refs, + - * / parens, registered calls)
   · FunctionRegistry (stub: min/max/clamp)
   · topological resolution of param-to-param refs; cycle detection
        │  (document with all expressions replaced by concrete numbers)
        ▼
 DocumentBuilder >> Builder >> OpRegistry dispatch (unchanged loop)
   · SketchOp (now rectangle | circle | polygon)
   · ExtrudeOp (unchanged) · PocketOp · HoleOp · FilletOp · ChamferOp · BooleanOp
        │
        ▼
 Kernel (build123d): new primitives + the OCCT boolean-robustness ladder
```

### New / changed units (one class per module)

- **`params/param_resolver.py` : `ParamResolver`**: `resolve_document(document: dict)
  -> dict` returns a copy with `parameters` fully evaluated and every `"${...}"` string
  value replaced by a number. Uses `ast.parse(expr, mode="eval")` and walks the tree in
  a restricted evaluator; anything outside the allowed node set raises `ExpressionError`.
- **`params/function_registry.py` : `FunctionRegistry`**: maps a name to a pure Python
  callable; `.with_defaults()` registers a small stub set (`min`, `max`, `clamp`). The
  evaluator dispatches `Call` nodes here; an unknown function raises `ExpressionError`.
- **`params/expression_error.py` : `ExpressionError(Exception)`**: a typed contract
  error (the document is malformed), raised at resolve time.
- **`ops/pocket_op.py` : `PocketOp`**, **`ops/hole_op.py` : `HoleOp`**,
  **`ops/fillet_op.py` : `FilletOp`**, **`ops/chamfer_op.py` : `ChamferOp`**,
  **`ops/boolean_op.py` : `BooleanOp`**: each `build(shape_in, params, provenance_in,
  kernel) -> OpResult`, registered in `OpRegistry.with_defaults()`.
- **`ops/sketch_op.py` : `SketchOp`** (extend): support `circle` and `polygon`
  elements alongside `rectangle`.
- **`ops/edge_selector.py` : `EdgeSelector`**: resolves an edge-set keyword against a
  solid's edges (via kernel edge metadata). Shared by `FilletOp`/`ChamferOp`. This is
  the deliberate seed of bucket 0.3's attribute model, kept to five keywords.
- **`kernel/kernel.py` / `build123d_kernel.py` / `tests/kernel/fake_kernel.py`**: new
  primitives (below); the build123d kernel wraps the fragile ops with the robustness
  ladder.

### Kernel interface additions

Behind the `Kernel` ABC, implemented in `Build123dKernel`, analytically faked in
`FakeKernel` (enough to assert op behaviour without OCP):

- `circle_face(center: Point2, diameter: float, plane: str) -> Any`
- `cylinder(center: Point3, axis: str, diameter: float, length: float) -> Any`
- `cut(solid, tools: list) -> Any` · `fuse(solids: list) -> Any` · `intersect(solids: list) -> Any`
- `fillet_edges(solid, edges: list, radius: float) -> Any`
- `chamfer_edges(solid, edges: list, distance: float) -> Any`
- `edges_of(solid) -> list` where each edge exposes enough metadata (orientation:
  `"vertical"|"horizontal"`, midpoint z) for `EdgeSelector`. Minimal by design; the full
  attribute model is bucket 0.3.

`polygon_face` (bucket 0.1) is reused for polygon sketch elements.

### Boolean-robustness ladder

`cut`/`fuse`/`intersect`/`fillet_edges`/`chamfer_edges` are the fragile OCCT ops
(`docs/research/occt-boolean-robustness.md`). The build123d kernel wraps each with the
documented ladder: normalize/validity-gate inputs (`BRepCheck_Analyzer`,
`ShapeFix_ShapeTolerance`) → fuzzy retry (`SetNonDestructive` + escalating
`SetFuzzyValue`, auto-tuned to bbox scale) → `UnifySameDomain` cleanup → validate +
`ShapeFix_Shape` heal-and-retry; on final failure it raises a typed kernel error the op
turns into an id-tagged `BuildIssue`.

## Expression model (design section 1)

HOCON carries three things, exactly per design section 1:
1. literals (`8`, `"mm"`),
2. parameter references + arithmetic (`margin = "${hole_d} * 1.5"`),
3. calls to registered functions (`clamp(${x}, 0, 10)`).

`ParamResolver` runs before feature dispatch. `${name}` refers to a key in the
document's `parameters`; refs may chain (a param referencing another), resolved in
topological order with cycle detection. `"${...}"` may also appear as a feature field
value (e.g. `"distance": "${thickness}"`); the resolver inlines those too. Values remain
numbers in the document's declared `units`; canonical mm conversion happens at kernel
export as today. No unit algebra in this bucket.

Allowed AST nodes: `Expression`, `Constant`/`Num`, `Name` (a bare param name, produced
by substituting `${x}` -> `x`), `BinOp` with `Add|Sub|Mult|Div`, `USub` (unary minus),
`Call` with a `Name` func resolved via the registry, and parentheses (grouping). Any
other node (attribute access, subscript, comprehension, lambda, name not in scope)
raises `ExpressionError`. This gives the design's "restricted AST, no filesystem /
network / global state".

## Data shapes (document)

```jsonc
{
  "schema_version": 2, "units": "mm",
  "parameters": { "w": 80, "h": 60, "thickness": 8, "hole_d": 6,
                  "margin": "${hole_d} * 1.5" },
  "parts": {
    "bracket": {
      "profile": "solid",
      "features": [
        { "id": "sk", "op": "sketch", "plane": "XY",
          "elements": [ { "id": "r", "type": "rectangle", "w": "${w}", "h": "${h}" } ] },
        { "id": "pad", "op": "extrude", "profile": "sk", "distance": "${thickness}" },
        { "id": "holes", "op": "hole", "plane": "XY", "diameter": "${hole_d}",
          "through": true,
          "positions": [ ["${margin}", "${margin}"], ["${w}-${margin}", "${margin}"],
                         ["${margin}", "${h}-${margin}"], ["${w}-${margin}", "${h}-${margin}"] ] },
        { "id": "soften", "op": "fillet", "radius": 2, "edges": "vertical" }
      ]
    }
  }
}
```

(Positions are illustrative; whether each coordinate can itself be an expression is a
plan detail: yes, each numeric field, including list elements, is resolved.)

## Schema changes (`part_schema.hocon`, permissive)

- `sketch_element.type` enum gains `circle`, `polygon`; add fields `r`, `d`, `points`
  (array of vec2), `sides` (integer). Keep existing `w`/`h`.
- `feature` gains op-specific optional fields: `positions` (array of vec2), `diameter`,
  `depth`, `through` (bool), `radius`, `edges` (enum `all|top|bottom|vertical|horizontal`).
  For the boolean op (whose feature `op` is `"boolean"`): `target` and `tool` (feature-id
  strings) and `operation` (enum `cut|union|intersect`). Numeric fields accept a number
  OR a `${...}` string (expressions), so their JSON-Schema type is `["number", "string"]`
  where an expression is allowed.
- No `additionalProperties: false`; only settled fields constrained.

## Provenance & error handling

- **Provenance** unchanged from bucket 0.1: `provenance[feature_id] = op_name`. Edge/face
  metadata for the fillet keyword is computed on demand from the live solid, not stored.
- **Expression errors** (unknown ref, disallowed syntax, unknown function, cycle) raise
  `ExpressionError` at resolve time; `DocumentBuilder` surfaces it naming the offending
  parameter/field. This is a contract error (malformed document), consistent with the
  "programmer errors raise; validation issues are data" rule.
- **Op failures** (empty edge set for a keyword, missing `target`/`tool`, kernel op
  failing after the robustness ladder) return `BuildIssue(node_id=feature_id, message=...)`,
  never a crash. The build continues and reports.

## Testing

- **Fast (FakeKernel, no OCP):**
  - `ParamResolver`: `${ref}` substitution; arithmetic with precedence and parens;
    chained param refs resolved topologically; cycle detection raises; unknown ref
    raises; disallowed syntax (attribute/subscript/lambda/import) raises; a registered
    function call evaluates; unknown function raises; expression as a feature field value.
  - `FunctionRegistry`: register/get; `with_defaults` has min/max/clamp; unknown raises.
  - Ops: `PocketOp` reduces volume; `HoleOp` drills N holes (volume/count); `BooleanOp`
    cut/union/intersect produce expected volumes; `SketchOp` circle/polygon areas;
    `EdgeSelector` maps each keyword to the right edge subset (via fake edge metadata);
    `FilletOp`/`ChamferOp` apply to the resolved subset and report an issue on an empty set.
  - `DocumentBuilder`: a parametric document resolves + builds; an expression error is
    raised with the offending name.
- **Slow (real build123d, `@pytest.mark.slow`):** the gate demo document builds, exports
  a non-empty glb, and exercises the boolean-robustness ladder (holes = cut, fillet).
- **Example + regression:** `examples/gate-0.2-bracket/bracket.hocon` (the demo) added to
  the gate-examples test so it is built and checked like gate-0.1 (fast via FakeKernel +
  the one slow real-kernel export).

## Boundaries / non-goals

- No patterns, counterbore/tapped holes, draft, thin walls, end-condition variety beyond
  blind/through (Phase 2/3).
- No real selector language or attribute model beyond the five edge keywords (bucket 0.3).
- No unit algebra; values are numbers in the document's unit.
- No multibody management; boolean is two-feature cut/union/intersect.
- Provenance is not deepened; the element map is bucket 0.3 / Phase 4.
